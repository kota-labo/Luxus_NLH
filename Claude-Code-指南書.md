# Claude Code 操作指南書 — ポーカーアプリ開発編

## 1. 基本の使い方

### 1.1 プロジェクトの開始

```bash
# 作業ディレクトリに移動して Claude Code を起動
cd /path/to/your/poker-app
claude

# または直接パスを指定
claude --cwd /path/to/your/poker-app
```

### 1.2 CLAUDE.md を最初に作る

プロジェクトルートに `CLAUDE.md` を置くと、Claude Code が毎回自動的に読み込む。
ポーカーアプリなら以下のような内容を書いておくと効率が上がる。

```markdown
# CLAUDE.md

## プロジェクト概要
テキサスホールデムのポーカーアプリ（Web / モバイル）

## 技術スタック
- フロントエンド: React + TypeScript
- バックエンド: Node.js / Express
- DB: PostgreSQL
- リアルタイム通信: WebSocket (Socket.IO)

## ドメイン用語
- ハンド: 各プレイヤーに配られる2枚のカード
- コミュニティカード: テーブル中央に開かれる5枚のカード
- ポット: ベットされたチップの合計
- アクション: フォールド / チェック / コール / レイズ / オールイン

## 開発ルール
- テストは `npm test` で実行
- lint は `npm run lint` で実行
- コミットメッセージは日本語OK
```

## 2. 効果的な指示の出し方

### 2.1 具体的に伝える（悪い例 → 良い例）

| 悪い例 | 良い例 |
|--------|--------|
| ポーカーのロジック作って | テキサスホールデムのハンド評価関数を作って。ロイヤルフラッシュからハイカードまで10段階でランク付けし、同ランク時のキッカー比較もサポートして |
| テスト書いて | `evaluateHand()` のユニットテストを書いて。フルハウス vs フラッシュ、同じツーペア同士のキッカー比較のケースを含めて |
| バグ直して | プリフロップでBBのプレイヤーがチェックできずにエラーになるバグを修正して。エラーログは `Cannot check: not enough bet` |

### 2.2 段階的に依頼する

ポーカーアプリのような複雑なプロジェクトは、一度に全部頼まず段階的に進める。

```
Step 1: 「カードのデッキとシャッフルの仕組みを作って」
Step 2: 「ハンド評価ロジックを作って」
Step 3: 「ベッティングラウンドのステートマシンを作って」
Step 4: 「WebSocketでマルチプレイヤー対応にして」
Step 5: 「UIを作って」
```

### 2.3 ドメイン知識を共有する

Claude はポーカーの基本ルールを知っているが、
ハウスルール・バリエーションがある場合は明示する。

```
うちのアプリでは以下のルールを採用している：
- ブラインド構造: SB=50, BB=100 から開始、10分ごとに倍増
- レイクなし（フリーミアム課金モデル）
- ショートデッキ（6+ホールデム）にも対応
- フラッシュがフルハウスより強い（ショートデッキルール）
```

## 3. ポーカー開発でよく使うコマンドパターン

### 3.1 コード生成

```
テキサスホールデムの手札評価エンジンをTypeScriptで実装して。
以下の要件：
- 7枚のカードから最強の5枚の組み合わせを選ぶ
- ハンドランクの列挙型を定義
- 2つのハンドを比較する関数を提供
- パフォーマンスを考慮（ルックアップテーブル推奨）
```

### 3.2 テスト生成

```
ハンド評価のエッジケースをテストして：
- Aを含むストレート（A-2-3-4-5 と 10-J-Q-K-A）
- 同じランクのハンド同士のタイブレーク
- コミュニティカード5枚がフラッシュの場合
```

### 3.3 バグ修正

```
以下のバグを調査して修正して：
再現手順:
1. 3人でゲーム開始
2. プリフロップでSBがフォールド
3. BBのターンが来ない（フリーズ）
関連ファイル: src/game/bettingRound.ts
```

### 3.4 リファクタリング

```
src/game/gameState.ts のゲーム状態管理を
有限ステートマシンパターンにリファクタして。
状態: WAITING → PREFLOP → FLOP → TURN → RIVER → SHOWDOWN → COMPLETE
```

## 5. 開発フロー例

### 典型的な1セッションの流れ

```
1. claude 起動
2. 「今日はベッティングロジックを実装する」と伝える
3. 設計について相談 → 実装方針を決める
4. 「実装して」で段階的にコードを書かせる
5. 「テスト書いて」でテストを作成
6. 「テスト実行して」で確認
7. 問題があれば修正を依頼
8. /commit でコミット
```

### Plan モードの活用

大きな機能を実装する前に、設計計画を立てさせる。

```
Plan モードで、マルチテーブルトーナメント機能の
実装計画を立てて。以下を考慮：
- テーブルバランシング（プレイヤー数の均等化）
- ブラインドレベルの自動進行
- バブル時のハンド・フォー・ハンド
- ファイナルテーブルへの集約
```

## 6. Tips & ベストプラクティス

### 6.1 やるべきこと

- **CLAUDE.md を育てる**: プロジェクトが進むたびにドメイン用語やルールを追記する
- **テストを先に書かせる**: ポーカーはエッジケースが多い。TDDが特に有効
- **エラーメッセージを貼る**: バグ報告時はスタックトレースをそのまま貼る
- **既存コードを読ませてから修正させる**: 「このファイルを読んで問題点を指摘して」
- **段階的に確認する**: 大きな実装は途中で動作確認を挟む

### 6.2 やってはいけないこと

- **曖昧な指示を出す**: 「いい感じにして」は避ける
- **一度に全部頼む**: ゲーム全体を一回の指示で作ろうとしない
- **確認せず進める**: 生成されたコードのロジックは必ずレビューする
- **セキュリティを後回しにする**: 乱数生成は `crypto.getRandomValues()` 等を使う（`Math.random()` はNG）
- **コンテキストを無駄にする**: 関係ない話題を混ぜない。話題が変わったら `/clear`

### 6.3 ポーカー特有の注意点

| 注意点 | 理由 |
|--------|------|
| 乱数の品質 | `Math.random()` は予測可能。暗号学的乱数を使う |
| 浮動小数点 | チップ計算は整数で行う（ポットの端数問題を防ぐ） |
| 状態の一貫性 | ベッティングラウンド中の状態遷移は厳密に管理する |
| タイミング攻撃 | カード情報の送信タイミングで手札が推測されないようにする |
| 切断対応 | プレイヤーが切断した場合のタイムアウト処理を必ず実装する |

---

## 7. 困ったときの対処法

| 状況 | 対処 |
|------|------|
| Claude が的外れな回答をする | コンテキストが汚れている可能性 → `/clear` して再度指示 |
| 生成コードがコンパイルエラー | エラーメッセージをそのまま貼って「このエラーを修正して」 |
| 複雑すぎて一度に実装できない | 機能を分割して段階的に依頼する |
| ポーカールールの解釈が違う | CLAUDE.md にルールを明記する |
| 同じミスを繰り返す | 「このパターンは避けて」と明示的に伝える（Claude が記憶する） |
