# CLAUDE.md — ポーカーアプリ開発プロジェクト

## プロジェクト概要

テキサスホールデムを中心としたポーカーアプリの開発プロジェクト。

## ドメイン用語

- **ハンド (Hand)**: 各プレイヤーに配られる2枚のカード（ホールカード）
- **コミュニティカード (Community Cards)**: テーブル中央に公開される最大5枚のカード
- **ポット (Pot)**: そのハンドでベットされたチップの合計
- **ブラインド (Blinds)**: SB（スモールブラインド）/ BB（ビッグブラインド）の強制ベット
- **アクション**: フォールド / チェック / コール / ベット / レイズ / オールイン
- **ポジション**: BTN（ボタン）/ SB / BB / UTG / MP / CO
- **ストリート**: プリフロップ → フロップ → ターン → リバー
- **ショーダウン**: リバー後に残ったプレイヤーが手札を公開し勝者を決定

## ハンドランク（強い順）

1. ロイヤルフラッシュ
2. ストレートフラッシュ
3. フォーオブアカインド（クワッズ）
4. フルハウス
5. フラッシュ
6. ストレート
7. スリーオブアカインド（セット / トリップス）
8. ツーペア
9. ワンペア
10. ハイカード

## ゲーム状態遷移

```
WAITING → PREFLOP → FLOP → TURN → RIVER → SHOWDOWN → COMPLETE
```

各ストリートではベッティングラウンドが完了してから次に進む。

## コーディングルール

### セキュリティ（必須）

- カードのシャッフル・配布には**暗号学的乱数**を使うこと（`crypto.getRandomValues()` / `crypto.randomBytes()` 等）
- `Math.random()` はカード関連処理では**絶対に使用禁止**
- サーバーサイドでカード情報を管理し、クライアントには見えるべきカードのみ送信する
- カード情報の送信タイミングでハンドが推測されないよう注意する

### データ型

- チップ量・ポット計算は**整数型（number の整数値）**で扱う。浮動小数点は使わない
- サイドポットの分配時は端数チップの処理ルールを明確にする（ポジション順で1チップずつ配分）

### 状態管理

- ゲーム状態は**有限ステートマシン**で管理する
- 各状態で許可されるアクションを厳密に定義する
- 不正な状態遷移は例外として処理する

### 切断・タイムアウト

- プレイヤーの切断時はタイムバンクを消費し、時間切れでフォールド扱いとする
- アクションのタイムアウトは必ず実装する

## テスト方針

- ハンド評価ロジックは**エッジケースを重点的に**テストする
  - A を含むストレート（A-2-3-4-5 と 10-J-Q-K-A）
  - 同ランクのタイブレーク（キッカー比較）
  - コミュニティカード5枚だけで役が成立する場合
  - サイドポットが複数発生する場合
- TDD を推奨: テストを先に書き、実装はテストを通すために書く

## コマンド

- テスト実行: `npm test`
- リント: `npm run lint`
- ビルド: `npm run build`
- 開発サーバー: `npm run dev`

## コミットルール

- コミットメッセージは日本語で記述してよい
- 機能単位で小さくコミットする

## 実装時の注意

- 大きな機能は段階的に実装する（デッキ → ハンド評価 → ベッティング → マルチプレイヤー → UI）
- 既存コードを必ず読んでから修正する
- 依頼された範囲のみ実装し、不要な拡張を加えない
